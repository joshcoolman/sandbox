HTMX vs React: The honest verdict (with receipts)!

Jul 6, 2026 https://www.youtube.com/watch?v=k1TrnzZaygo

This is a button. You click it and the number changes. Fresh from the server, no page reload. Now, here's the strange part: there is no JavaScript here — none that you wrote. The whole thing is one HTML attribute. And one real company used this idea to delete 67% of their React code. So, is dumb HTML really killing React? Let's settle it.

To make that one number update the modern way, you'd reach for React: a component, a piece of state, a fetch call, a JSON endpoint on the server, and a re-render to stitch it all back together. That is a lot of moving parts for one number on a screen. HTMX throws all of it out. The secret is almost insultingly simple: let plain HTML talk to the server directly and swap in whatever HTML comes back. Dumb, and it works. But the internet is split down the middle on whether that is genius or a step backwards. By the end of this, you will know exactly when to reach for it and when it will burn you.

Start with the weight. HTMX is about 16 kilobytes, zero dependencies, plain JavaScript you never have to touch. React plus React DOM, just the core before your app does anything, is roughly 45. And a real single page app ships hundreds of kilobytes before it shows you a thing. HTMX was made by Carson Gross, a computer science professor in Montana — not a trillion-dollar framework team. And the entire loop is three words: request, response, swap. You add an attribute, the element fires an HTTP request, the server sends back a chunk of HTML, and HTMX drops it into the page. That is it.

Now, let's actually break down the anatomy. hx-get or hx-post says where to send the request. hx-target says which element on the page should change. hx-swap says how — replace the insides, the whole element, or just add to it. An hx-trigger decides when it fires: a click, a key press, a form change, even a timer every couple of seconds for a real-time dashboard. Four attributes.

Here's the part that breaks the framework habit. It isn't just buttons — any element can make a request, any event can trigger it, every HTTP verb is on the table (get, post, put, delete — not just the two that HTML forms gave you), and you choose exactly which part of the page gets replaced. Any element, any event, any verb, any target.

So why does returning HTML instead of JSON matter so much? Because of an old idea with a fancy name: hypermedia. The server doesn't just send data — it sends the next set of actions baked right into the HTML. The page itself carries the state of your application. And here's the part that starts arguments: this is REST. The real REST, the way Roy Fielding actually defined it. Somewhere along the line, the word REST got hijacked to mean a JSON API. HTMX is just quietly going back. It also brings back something we lost: locality of behavior. Look at the element and you can see exactly what it does right there in the markup — no hunting through a dozen files to trace one click.

Which brings us to the receipt. A real company, Contact, rewrote their React app in HTMX. The result: 67% less code. Their JavaScript dependencies dropped from 255 down to nine. And build times fell by nearly 90%. This is no toy — 48,000 stars on GitHub, the number one rising star in front end, and in the big developer surveys, one of the single most admired tools in the entire industry.

In the React world, you keep state twice — once on the server, once in the browser — and write an API forever just to keep them in sync. HTMX keeps a single copy on the server and sends HTML. A whole category of bugs simply evaporates.

Because the server just needs to return HTML, your back end can be literally anything. The community even has a name for it: the Hower Stack — Hypermedia on whatever you'd like. Django, Flask, FastAPI, Rails, Laravel, Go, Express — pick your language, HTMX genuinely does not care. When you do need real client-side interactivity — a drop-down, a toggle, a small animation — you reach for a tiny partner like Alpine.js or Hyperscript. Small tools sprinkled in where you need them, instead of one giant framework running everything.

If this is reminding you of Rails Hotwire and Turbo, you're right — same instinct, different flavor. Turbo is more opinionated and batteries included; HTMX is lower level and does only what you explicitly tell it to.

Now, the comment section, because this is where the fight actually is. The most liked objection: isn't this just Ajax and jQuery reinvented? And honestly, yes, kind of — veterans who did this back in 2010 are watching the web close a full circle. HTMX's answer is that it took that old instinct and turned it into a clean, complete, declarative attribute system, not a pile of hand-wired scripts.

Objection two: "no JavaScript" is a lie — HTMX is JavaScript. Fair. The honest claim is no JavaScript that you write.

Objection three, the real technical one: every action is now a network request. On a slow connection, a snappy app can crawl. That is a genuine trade-off, and it's exactly why those tiny partner libraries exist. This isn't magic — it's a different set of trade-offs.

So let's be honest, because HTMX fans often aren't: this is the wrong tool for some apps. A spreadsheet like Google Sheets, a map you drag in real time, anything offline or with thousands of interdependent widgets reacting instantly — that is React's home turf, and it isn't close. And the skeptics have a point on adoption too: you'll see HTMX all over solo projects and side projects, and far less inside big enterprise front ends, for now.

So no, this is not React's funeral. If you're building a deeply interactive client application, reach for React — that is exactly what it's brilliant at. HTMX is for the other 90% of the web: forms, dashboards, content CRUD — the boring stuff that secretly makes up most of everything.

So, to recap: HTMX is about 16 kilobytes that let HTML talk to the server. It swaps in HTML instead of juggling JSON. It revives real hypermedia and real REST. And for most apps, it deletes a staggering amount of complexity. The team now calls the 2.x line feature complete, and HTMX 4 is already in beta, folding in DOM morphing and streaming responses. But the pitch hasn't changed since day one: do more by writing far less.

So here's the real question, the one the whole internet is arguing about: is HTMX a genuine breakthrough, or just jQuery with better marketing?
