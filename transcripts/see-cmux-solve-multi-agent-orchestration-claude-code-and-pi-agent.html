The big ideas:

  • Scale your compute to scale your impact — the more agents you can run and actually oversee, the more your engineering output scales.
  • Three real problems multi-agent orchestration has to solve: no programmatic access to your agents, no visibility into what they're doing ("an agent you can't monitor is an agent you can't improve"), and manual team setup killing your speed.
  • Cmux's mental model is window → workspace → pane, controlled through a simple send-key / read-screen / open-close-surface API — agentic access to every terminal, not just human access.
  • Favorite pattern: three-tier orchestration — an orchestrator prompts team leads, leads prompt specialized worker agents — with flat, not top-down, communication, so any agent can prompt any other agent directly.
  • Demonstrated at scale: agent fleets in grid layouts, agents paired with in-app browsers, and an "8-agent hot-fix race" pattern — throw several different models/tools at the same production bug in parallel and ship whichever solves it first.
  • Cmux vs. tmux: Cmux is Mac-only, newer, and more customizable/aesthetic with agentic-access-first design; tmux is the mature, cross-platform original. Verdict is a cautious "fan so far," watching for rough edges as the tool matures.
  • Daily workflow: a one-command ("just file") launcher instantly boots a full 3-tier agent team scoped to a named feature, instead of hand-assembling panes every time.
